What is Jenkins?

Jenkins is an open-source, Java-based automation server used to automate the build, test, and deployment parts of software development. It is the most widely used CI/CD tool in DevOps and integrates with almost every tool in the ecosystem.

Open Source

Free, community-driven. 1,800+ plugins available for every use case.

Java-based

Requires JDK 11/17. Runs as a war file or system service on any OS.

Plugin Ecosystem

Git, Docker, Kubernetes, AWS, Slack — pluggable for any workflow.

Pipeline as Code

Define your entire CI/CD workflow in a Jenkinsfile stored in your repo.

Master–Agent

Distribute builds across multiple agents — Linux, Windows, Docker, cloud.

Triggers

GitHub webhooks, SCM polling, cron schedules, or manual triggers.

Jenkins vs GitHub Actions: Jenkins is self-hosted, fully customizable, and free at scale. GitHub Actions is cloud-native and simpler. Jenkins is preferred when you need full control, on-prem infra, or complex multi-stage pipelines.

Jobs & Builds

A Job (or Item) is a runnable task in Jenkins. Each execution of a job is called a Build, numbered incrementally (#1, #2, #3…).

Freestyle Project

GUI-configured job. Good for simple shell scripts and basic pipelines.

Pipeline

Code-defined job using a Jenkinsfile. The standard for production.

Multibranch Pipeline

Auto-creates pipelines for each branch in a Git repo.

Folder

Organizes jobs into groups. Each folder has its own credentials and views.

Build ConceptDescription
#BUILD_NUMBERAuto-incremented integer for each run of a job
ArtifactFile produced by a build (JAR, Docker image, ZIP) and saved for later use
WorkspaceTemp directory on the agent where the build runs and files are checked out
Upstream / DownstreamJob A triggers Job B — A is upstream, B is downstream
ParameterUser-provided input to a build (string, boolean, choice, credentials)

Jenkinsfile

A Jenkinsfile is a text file stored in the root of your Git repository that defines the entire CI/CD pipeline as code. This enables versioning, code review, and reproducibility.

Minimal Declarative Jenkinsfile
Jenkinsfilepipeline {
  agent any             // run on any available agent

  stages {
    stage('Checkout') {
      steps {
        git branch: 'main', url: 'https://github.com/user/repo.git'
      }
    }
    stage('Build') {
      steps {
        sh 'npm install'
      }
    }
    stage('Test') {
      steps {
        sh 'npm test'
      }
    }
    stage('Deploy') {
      steps {
        sh './deploy.sh'
      }
    }
  }

  post {
    success { echo 'Pipeline succeeded!' }
    failure { echo 'Pipeline failed!' }
    always  { cleanWs() }   // clean workspace after every run
  }
}
Where to put it: File named exactly Jenkinsfile (capital J, no extension) in the root of your repository. Jenkins finds it automatically when you configure the Pipeline job.

Declarative Pipeline — Full Reference

Full Declarative Syntax
pipeline {
  agent { label 'linux' }   // or: any, none, docker{}

  environment {             // pipeline-wide env vars
    APP_NAME = 'my-app'
    DOCKER_REGISTRY = 'docker.io/myuser'
  }

  options {
    timeout(time: 30, unit: 'MINUTES')
    buildDiscarder(logRotator(numToKeepStr: '10'))
    disableConcurrentBuilds()
  }

  parameters {
    string(name: 'DEPLOY_ENV', defaultValue: 'staging', description: 'Target env')
    booleanParam(name: 'RUN_TESTS', defaultValue: true)
  }

  stages {
    stage('Build') {
      steps {
        sh 'mvn clean package -DskipTests'
      }
    }

    stage('Test') {
      when { expression { params.RUN_TESTS == true } }
      steps {
        sh 'mvn test'
        junit '**/target/surefire-reports/*.xml'
      }
    }

    stage('Deploy') {
      when { branch 'main' }
      steps {
        sh "./deploy.sh ${params.DEPLOY_ENV}"
      }
    }
  }

  post {
    always   { archiveArtifacts artifacts: '**/target/*.jar' }
    failure  { mail to: 'team@co.com', subject: "FAILED: ${env.JOB_NAME}" }
    success  { echo 'All good!' }
    unstable { echo 'Tests are failing!' }
  }
}
BlockPurpose
agentWhere the pipeline/stage runs (any, none, label, docker)
environmentDefine env vars accessible as env.VAR_NAME throughout
optionsPipeline behaviour — timeouts, log rotation, concurrent builds
parametersInput values users can provide before running the build
stagesContainer for all stage blocks
stage('name')A logical step — shown as a column in the pipeline view
stepsThe actual commands/actions inside a stage
whenConditional execution — branch name, env var, expression, etc.
postActions after pipeline finishes — always, success, failure, unstable, changed

Scripted Pipeline

Scripted pipelines use full Groovy syntax inside a node{} block. More powerful but more complex — good for advanced conditional logic and loops.

Scripted Pipeline Example
node('linux') {               // run on agent with label 'linux'
  def appName = 'my-app'
  def version = ''

  stage('Checkout') {
    checkout scm
    version = sh(script: 'git rev-parse --short HEAD', returnStdout: true).trim()
  }

  stage('Build') {
    sh "docker build -t ${appName}:${version} ."
  }

  try {
    stage('Test') {
      sh 'npm test'
    }
  } catch (err) {
    echo "Tests failed: ${err}"
    currentBuild.result = 'UNSTABLE'
  }

  if (env.BRANCH_NAME == 'main') {
    stage('Deploy') {
      sh "./deploy.sh ${version}"
    }
  }
}

Agents & Nodes

Jenkins uses a Controller (master) to schedule jobs and one or more Agents (workers) to run them. This enables distributed, parallel builds.

Agent Directive Options
// Run on any available agent
agent any

// Run on controller (not recommended for heavy builds)
agent none

// Run on agent with a specific label
agent { label 'linux && docker' }

// Run inside a Docker container
agent {
  docker {
    image 'node:18-alpine'
    args  '-v /tmp:/tmp'
  }
}

// Per-stage agent (when pipeline agent is none)
stage('Build') {
  agent { label 'build-server' }
  steps { sh 'make build' }
}
Adding an Agent: Manage Jenkins → Manage Nodes → New Node → Set labels, Remote root, Launch method (SSH or JNLP). Install Java on the agent first.

Parallel Execution

Run multiple stages simultaneously to cut pipeline time — e.g., run Unit Tests and Integration Tests at the same time.

Parallel Stages
stage('Test') {
  parallel {
    stage('Unit Tests') {
      agent { label 'linux' }
      steps { sh 'npm run test:unit' }
    }
    stage('Integration Tests') {
      agent { label 'linux' }
      steps { sh 'npm run test:integration' }
    }
    stage('Lint') {
      agent { label 'linux' }
      steps { sh 'npm run lint' }
    }
  }
}

// Fail fast: stop all parallel branches if one fails
stage('Parallel Tests') {
  failFast true
  parallel { ... }
}

Docker Integration

Pre-requisite: Docker must be installed on the Jenkins agent. Add the jenkins user to the docker group: sudo usermod -aG docker jenkins, then restart Jenkins.
Build & Push Docker Image in Pipeline
pipeline {
  agent any
  environment {
    DOCKER_IMAGE = "myuser/my-app:${BUILD_NUMBER}"
  }
  stages {
    stage('Build Image') {
      steps {
        sh "docker build -t ${DOCKER_IMAGE} ."
      }
    }
    stage('Push to DockerHub') {
      steps {
        withCredentials([usernamePassword(
          credentialsId: 'dockerhub-creds',
          usernameVariable: 'DOCKER_USER',
          passwordVariable: 'DOCKER_PASS'
        )]) {
          sh """
            echo $DOCKER_PASS | docker login -u $DOCKER_USER --password-stdin
            docker push ${DOCKER_IMAGE}
            docker logout
          """
        }
      }
    }
    stage('Deploy Container') {
      steps {
        sh """
          docker stop my-app || true
          docker rm   my-app || true
          docker run -d --name my-app -p 8000:8000 ${DOCKER_IMAGE}
        """
      }
    }
  }
  post {
    always { sh "docker rmi ${DOCKER_IMAGE} || true" }
  }
}

Using Credentials in Jenkinsfile
// Secret text
withCredentials([string(credentialsId: 'slack-token', variable: 'SLACK_TOKEN')]) {
  sh 'curl -X POST -d "token=$SLACK_TOKEN" ...'
}

// Secret file (e.g. kubeconfig)
withCredentials([file(credentialsId: 'kubeconfig', variable: 'KUBECONFIG_FILE')]) {
  sh 'kubectl --kubeconfig=$KUBECONFIG_FILE get pods'
}

// SSH Key
withCredentials([sshUserPrivateKey(
  credentialsId: 'prod-ssh',
  keyFileVariable: 'SSH_KEY',
  usernameVariable: 'SSH_USER'
)]) {
  sh 'ssh -i $SSH_KEY $SSH_USER@prod-server.com "docker restart app"'
}
Never hardcode secrets in Jenkinsfiles. Jenkins masks credential values in logs automatically when using withCredentials, but they must be stored in the Credentials store first.

Shared Libraries

Shared Libraries let you define reusable Groovy code in a separate Git repo and import it into any Jenkinsfile. Prevents copy-paste across pipelines.

Library Repo Structure
jenkins-shared-library/
├── vars/
│   ├── buildAndPush.groovy   // callable as buildAndPush()
│   └── deployToK8s.groovy    // callable as deployToK8s()
└── src/
    └── org/
        └── myapp/
            └── Utils.groovy  // importable class
vars/buildAndPush.groovy
def call(String imageName, String tag) {
  sh """
    docker build -t ${imageName}:${tag} .
    docker push ${imageName}:${tag}
  """
}
Using the shared library in Jenkinsfile
// Register in: Manage Jenkins → Configure System → Global Pipeline Libraries
// Name: my-shared-lib  |  Default version: main  |  SCM: Git

@Library('my-shared-lib') _   // import everything

pipeline {
  agent any
  stages {
    stage('Build & Push') {
      steps {
        buildAndPush('myuser/my-app', env.BUILD_NUMBER)
      }
    }
  }
}

Multibranch Pipelines

Multibranch Pipeline jobs automatically discover branches (and PRs) in your repo and create a pipeline for each branch that has a Jenkinsfile.

Setup
// New Item → Multibranch Pipeline
// Branch Sources: Add source → GitHub
// Set repository URL + credentials
// Scan Multibranch Pipeline Triggers: 1 minute (optional)

// Jenkins will auto-discover branches and run Jenkinsfile in each one
Jenkinsfile with branch-based logic
pipeline {
  agent any
  stages {
    stage('Build')  { steps { sh 'make build' } }
    stage('Test')   { steps { sh 'make test'  } }

    stage('Deploy Staging') {
      when { branch 'develop' }
      steps { sh './deploy.sh staging' }
    }

    stage('Deploy Production') {
      when { branch 'main' }
      steps {
        input 'Approve deploy to production?'
        sh './deploy.sh production'
      }
    }
  }
}

Project 1 — Simple Node.js CI Pipeline

Project 01 / Beginner

Automated CI for a Node.js application

Set up a complete Jenkins pipeline that triggers on every GitHub push, installs dependencies, runs tests, and archives build artifacts. The foundation of all CI workflows.

Node.jsGitHub Webhooknpm testArtifacts
Jenkinsfile
pipeline {
  agent any

  tools {
    nodejs 'NodeJS-18'   // configured in Global Tool Configuration
  }

  stages {
    stage('Checkout') {
      steps { checkout scm }
    }
    stage('Install') {
      steps { sh 'npm install' }
    }
    stage('Lint') {
      steps { sh 'npm run lint' }
    }
    stage('Test') {
      steps {
        sh 'npm test -- --reporter=junit --reporter-options mochaFile=results/test-results.xml'
      }
      post {
        always {
          junit 'results/test-results.xml'
        }
      }
    }
    stage('Build') {
      steps {
        sh 'npm run build'
        archiveArtifacts artifacts: 'dist/**', fingerprint: true
      }
    }
  }
  post {
    always { cleanWs() }
  }
}

Project 2 — Docker CI/CD Pipeline

Project 02 / Intermediate

Full CI/CD with Docker Build → Push → Deploy

Build a Docker image on every push, push it to DockerHub with a versioned tag, then SSH into the production server and restart the container with the new image. The classic production CI/CD flow.

DockerDockerHubSSH DeployVersioned Tags
Jenkinsfile
pipeline {
  agent any

  environment {
    IMAGE_NAME  = "myuser/wanderlust-app"
    IMAGE_TAG   = "${BUILD_NUMBER}"
    FULL_IMAGE  = "${IMAGE_NAME}:${IMAGE_TAG}"
  }

  stages {
    stage('Checkout') {
      steps { checkout scm }
    }

    stage('Build Image') {
      steps {
        sh "docker build -t ${FULL_IMAGE} ."
      }
    }

    stage('Push to DockerHub') {
      steps {
        withCredentials([usernamePassword(
          credentialsId: 'dockerhub-creds',
          usernameVariable: 'DOCKER_USER',
          passwordVariable: 'DOCKER_PASS'
        )]) {
          sh """
            echo $DOCKER_PASS | docker login -u $DOCKER_USER --password-stdin
            docker push ${FULL_IMAGE}
            docker tag  ${FULL_IMAGE} ${IMAGE_NAME}:latest
            docker push ${IMAGE_NAME}:latest
          """
        }
      }
    }

    stage('Deploy to Server') {
      steps {
        withCredentials([sshUserPrivateKey(
          credentialsId: 'prod-ssh-key',
          keyFileVariable: 'SSH_KEY'
        )]) {
          sh """
            ssh -o StrictHostKeyChecking=no -i $SSH_KEY ubuntu@prod-server.com \
              "docker pull ${FULL_IMAGE} && \
               docker stop app || true && \
               docker rm   app || true && \
               docker run -d --name app -p 80:3000 ${FULL_IMAGE}"
          """
        }
      }
    }
  }

  post {
    always {
      sh "docker rmi ${FULL_IMAGE} || true"
      cleanWs()
    }
  }
}

Project 3 — Full Stack CI/CD to AWS EC2

Project 03 / Advanced

Production-grade pipeline — Code → Docker → ECR → EC2

Push the image to AWS ECR (instead of DockerHub), deploy to an EC2 instance using AWS credentials, send Slack notification on success or failure. Covers real-world production patterns.

AWS ECREC2 DeployAWS CLISlackSonarQube
Jenkinsfile (Production Pattern)
pipeline {
  agent any

  environment {
    AWS_REGION     = 'ap-south-1'
    ECR_REGISTRY   = '123456789.dkr.ecr.ap-south-1.amazonaws.com'
    ECR_REPO       = 'my-app'
    IMAGE_TAG      = "${BUILD_NUMBER}"
    FULL_IMAGE     = "${ECR_REGISTRY}/${ECR_REPO}:${IMAGE_TAG}"
  }

  stages {
    stage('Checkout') {
      steps { checkout scm }
    }

    stage('SonarQube Analysis') {
      steps {
        withSonarQubeEnv('sonarqube') {
          sh 'mvn sonar:sonar'
        }
      }
    }

    stage('Quality Gate') {
      steps {
        timeout(time: 5, unit: 'MINUTES') {
          waitForQualityGate abortPipeline: true
        }
      }
    }

    stage('Build & Push to ECR') {
      steps {
        withCredentials([[
          $class: 'AmazonWebServicesCredentialsBinding',
          credentialsId: 'aws-creds',
          accessKeyVariable: 'AWS_ACCESS_KEY_ID',
          secretKeyVariable: 'AWS_SECRET_ACCESS_KEY'
        ]]) {
          sh """
            aws ecr get-login-password --region ${AWS_REGION} | \
              docker login --username AWS --password-stdin ${ECR_REGISTRY}
            docker build -t ${FULL_IMAGE} .
            docker push  ${FULL_IMAGE}
          """
        }
      }
    }

    stage('Deploy to EC2') {
      steps {
        withCredentials([sshUserPrivateKey(
          credentialsId: 'ec2-ssh',
          keyFileVariable: 'EC2_KEY'
        )]) {
          sh """
            ssh -i $EC2_KEY ec2-user@ec2-xx-xx.ap-south-1.compute.amazonaws.com \
              "aws ecr get-login-password --region ${AWS_REGION} | \
               docker login --username AWS --password-stdin ${ECR_REGISTRY} && \
               docker pull ${FULL_IMAGE} && \
               docker stop app || true && docker rm app || true && \
               docker run -d --name app -p 80:8080 ${FULL_IMAGE}"
          """
        }
      }
    }
  }

  post {
    success {
      slackSend channel: '#ci-cd', color: 'good',
        message: "✅ Deployed ${env.FULL_IMAGE} to production\n${env.BUILD_URL}"
    }
    failure {
      slackSend channel: '#ci-cd', color: 'danger',
        message: "❌ Build FAILED: ${env.JOB_NAME} #${env.BUILD_NUMBER}\n${env.BUILD_URL}"
    }
    always { cleanWs() }
  }
}

Best Practices

  • Store Jenkinsfile in Git — never configure pipelines through the UI alone. Code is the source of truth.
  • Use Declarative pipelines — they're more readable, easier to audit, and enforce structure.
  • Never hardcode secrets — always use the Credentials store and withCredentials{}.
  • Set timeouts — add timeout(time: 30, unit: 'MINUTES') to prevent hung builds consuming agents forever.
  • Clean workspaces — always call cleanWs() in the post { always {} } block.
  • Use agents for heavy work — don't run builds on the controller. Keep the controller for scheduling only.
  • Tag Docker images with BUILD_NUMBER — never overwrite :latest only. Versioned tags enable rollback.
  • Add quality gates — integrate SonarQube and fail the pipeline if code quality drops below threshold.
  • Use Shared Libraries — once you have 3+ repos with similar Jenkinsfiles, extract common logic.
  • Lock production deploys — use input() for manual approval before deploying to production.
  • Monitor Jenkins itself — set up disk space alerts, plugin update notifications, and log monitoring.
